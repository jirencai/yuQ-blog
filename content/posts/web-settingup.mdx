# 利用vibe-coding 建立起来的小站

## 结论先行
利用三个小时和 Codex 以及 ChatGPT 进行交流，并且零代码投入完成了这个小站。
下面总结一下在创建这个小站的时候遇到的问题与感想。

## 关键点
- 前言
- 过程

## 前言
在产生设计自己博客这个想法之前，我没有任何关于前端、后端以及网站搭建的基础。这完全是个从 0 开始的项目，一开始的目的也仅仅是验证 Codex 的高效性。

一开始我是看到了 LanternCX 大佬对于 vibe-coding 的理解，并且他说他也 vibe 了几个项目。我在两天以前对于 vibe 还没有任何的概念，也就是他带我进了 vibe-coding 的圈子，随后我就真正去了解了什么是 vibe-coding。然后我发现 vibe-coding 可能不需要自己具备多么强大的编程能力，这是一种利用自然语言转换到实际程序的行为，所以只需要我的自然语言足够流畅、表达清楚，我就可以构建出这个项目。

抱着这样的想法，我去查阅了现在 vibe-coding 有哪些 agent 可以使用。现在主流的 agent 主要是 Codex、Claude Code 以及 OpenCode。

在开始的时候，我听说 Claude Code 的代码处理能力非常强，于是我去查阅了如何利用 Claude Code 进行 vibe-coding。我跟着网上的教程下载了 Claude Code 也就是 CLI 的版本。甚至在最开始的时候，我无法理解什么是 CLI。下载之后，我让他对我以前的一个工程进行结构化分析，发现在没有输出特别多有用信息的时候，token 已经消耗了很多，所以有人说 Claude 贵不是没有道理的。

再后来我去 B 站上看大家对于 vibe-coding 的讨论。但是这个话题比较新，很多东西要么就是在实验阶段，要么就是更新迭代太快，已经被淘汰了。所以我找了个给新不新旧不旧的视频看了下，里面讲的是 Codex App，也就是我完成这个网站所利用的编程环境。

第二天我就下载了 Codex，依旧让他给我总结一下项目的结构。他梳理的结构也很清晰，不过因为他是利用了 ChatGPT，所以不怎么耗钱，还是比较友好的。

然后我试着让他去做一个项目，就是做一个科技树编排的项目。我跟他说了串 prompt，然后他就照着这个 prompt 去做，最后我发现他的项目结构都是揉在一起的，似乎缺乏了一些合理性。

随后我再去拜读了一下 LanternCX 大佬的文章，发现对于 agent 需要一个全局的规范，让他按照这个规范走才可以做出有结构的项目。

随后我又经历了一系列探索，像是安装使用了一段时间 Codex CLI，甚至去研究了一下 SKILL，使用了一下 superpower。然后我发现这些对于我这个新手来说可能还是有点超前了。应该要从一个实际的项目出发，将 Codex 的基本功能掌握一下，于是就有了这个博客。

## 过程
在设计这个网站的时候，我一开始也没有什么思路。但是我可以问，于是我问 ChatGPT 建立一个自己的博客需要哪些流程。GPT 跟我讲了一大堆，显然我是看不下去的，因为我完全不懂嘛。然后我就问他可不可以用 Codex 完成这些，他说可以然后给了我具体的方案。

首先要明白与 Codex 建立一个项目的具体流程是什么。一个好的思路是一个好的开头。我的想法是：我来输出观点和需求，ChatGPT 和我来实现提示词的输入和项目的验收。

建立一个 `AGENTS.md` 是很有必要的，这个是 Codex 执行每次任务的规范。

为了后期项目的可维护性和拓展性，需要在项目的初期就将项目的框架搭建好。所以一开始让 Codex 做的就是搭建项目的框架，随后一步步去实现功能。每完成一个功能，都需要让他输出这个阶段的日志和验收效果，这样可以形成一个闭环，让 Codex 完美地按照我们的预期来走，从而不会走偏。

`AGENTS.md` 是 Codex 需时刻遵守的行为规范，这里我的 `AGENTS.md` 是让 ChatGPT 直接生成的。

当我完成对于 Codex 的行为规范以后，做的第一件事不是去直接投入到项目的进行中，而是和 Codex 对齐我们的任务目标，将我们的整个项目划分成了几个小块，通过在每个小块设计验收点，来让 Codex 时刻遵循我们的计划。
