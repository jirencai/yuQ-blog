# PLANS.md

## 1. 当前项目理解（基于 `.agents/AGENTS.md`）

### 项目目标
这个项目是一个个人博客网站，目标是在第一版快速做出一个干净、现代、可维护、可上线，并且后续可安全迭代的产品。

### V1 / MVP 要做什么
第一版聚焦以下能力：

- Home 页面
- Blog Index 页面
- Blog Post Detail 页面
- About 页面
- 响应式布局
- 暗黑模式
- 基础 SEO
- 本地内容写作（MDX 优先，或 Markdown）
- 可部署到 Vercel

### V1 明确不做什么
第一版不引入以下内容（除非后续明确提出）：

- 传统后端服务
- 数据库
- 认证/登录
- 管理后台
- CMS
- 评论系统
- 全文搜索
- 多语言
- 复杂分析系统
- 过度工程化抽象

### 为什么当前阶段不需要后端
V1 的核心场景是“内容发布与阅读”，内容来源是本地 MDX/Markdown 文件，天然适配静态/预渲染模式。此阶段如果引入后端会增加开发、部署、维护成本，但对 MVP 核心价值提升有限。

### 为什么优先保持简单、可维护、可上线
- MVP 的关键是尽快交付可用版本，而不是功能广度。
- 简单结构能降低返工成本，减少隐藏耦合。
- 可维护代码能让后续迭代（比如迁移 CMS）更平滑。
- 可上线优先保证真实使用闭环，便于基于反馈演进。

## 2. 推荐项目结构方案

建议采用如下目录结构（V1 适配）：

```text
app/
  layout.tsx
  page.tsx
  blog/
    page.tsx
    [slug]/
      page.tsx
  about/
    page.tsx
components/
  ui/
  blog/
content/
  posts/
lib/
  posts/
  seo/
public/
styles/
```

### 页面层如何组织（`app/`）
- 仅负责路由和页面编排。
- 页面通过 `lib` 获取数据，不直接处理文件读取与解析。
- 用 App Router 清晰表达页面职责：`/`、`/blog`、`/blog/[slug]`、`/about`。

### 内容层如何组织（`content/posts/`）
- 所有博客正文和 frontmatter 放在 `content/posts/`。
- 页面组件不内嵌内容，保持内容与展示解耦。
- 内容文件使用稳定字段（如 `title`、`date`、`summary`、`tags`、`slug`）。

### 组件层如何组织（`components/`）
- `components/ui`：通用 UI（按钮、容器、排版块）。
- `components/blog`：博客域组件（文章卡片、文章头部、目录块等）。
- 组件只关注呈现与交互，不承担数据装载职责。

### 数据/工具逻辑如何组织（`lib/`）
- `lib/posts`：内容读取、解析、排序、查询（如 `getAllPosts` / `getPostBySlug`）。
- `lib/seo`：基础 metadata 构建工具。
- 在 `lib` 内定义稳定的内容类型接口（例如 `PostMeta`、`Post`）。

### 如何为未来 CMS / API / DB 留空间，但不提前引入复杂度
- 页面层只依赖 `lib/posts` 提供的统一接口，不依赖文件系统细节。
- V1 先实现“文件型数据源”适配器；未来若换 CMS/API，仅替换 `lib/posts` 内部实现。
- 不提前引入网络层、数据库 schema、鉴权、后台等复杂模块。

## 3. V1 数据流与内容流设计

### 数据流步骤
1. 文章来源：`content/posts/*.mdx`（或 `.md`）。
2. 解析处理：`lib/posts` 读取文件，解析 frontmatter + 正文，生成统一结构。
3. 数据提供：页面在构建/请求阶段调用 `lib/posts` 接口获取文章列表或详情。
4. 页面渲染：
   - `blog/page.tsx` 渲染文章列表；
   - `blog/[slug]/page.tsx` 渲染单篇正文；
   - `page.tsx`/`about/page.tsx` 组合静态内容与文章入口。

### 当前结构为何利于未来演进
- 页面不耦合数据来源，未来可切换到 CMS/API/DB。
- 内容模型统一后，后续新增字段与排序策略可局部演进。
- 分层清晰可控，避免“边做边耦合”导致的重构成本。

## 4. Milestones（执行拆分）

### Milestone 1：项目骨架 / 基础结构
- 目标：建立最小可运行工程与分层边界。
- 工作内容：
  - 初始化 Next.js + TypeScript + App Router + Tailwind。
  - 创建 `app/components/content/lib/public/styles` 基础目录。
  - 建立基础 layout、全局样式入口、路由占位页。
- Definition of Done：
  - 项目可启动、可构建。
  - 目录职责清晰，无明显跨层耦合。
- 验证方式：
  - 本地启动和构建通过。
  - 路由页面可访问。

### Milestone 2：博客基础页面和内容渲染
- 目标：打通“本地内容 -> 页面展示”主链路。
- 工作内容：
  - 实现 Home、Blog Index、Post Detail、About 页面。
  - 实现 `lib/posts` 的读取、解析、排序、slug 查询。
  - 添加至少 2 篇示例文章。
- Definition of Done：
  - 列表页可展示文章摘要并跳转详情页。
  - 详情页可按 slug 正确渲染正文。
- 验证方式：
  - 页面手动访问验证。
  - 关键路由与 slug 错误场景检查。

### Milestone 3：阅读体验优化
- 目标：提升阅读舒适度与跨端一致性。
- 工作内容：
  - 响应式排版优化（移动端优先）。
  - 暗黑模式支持与主题样式统一。
  - 正文样式优化（标题、段落、代码块、引用等）。
- Definition of Done：
  - 移动端与桌面端阅读体验稳定。
  - 明暗主题切换一致，无明显视觉断层。
- 验证方式：
  - 多断点手测。
  - 明暗主题切换走查。

### Milestone 4：SEO 和部署准备
- 目标：具备可发现性与可上线能力。
- 工作内容：
  - 配置基础 metadata（title/description/Open Graph）。
  - 准备 robots/sitemap 基础策略。
  - 完成 Vercel 部署配置与构建检查。
- Definition of Done：
  - 生产构建通过且可部署。
  - 页面具备基础 SEO 元信息。
- 验证方式：
  - 本地生产构建验证。
  - 部署预览环境检查页面头信息。

### Milestone 5：视觉优化和 polish
- 目标：在不扩展需求前提下提升完成度。
- 工作内容：
  - 优化视觉层级、间距、配色、状态反馈。
  - 修复主要页面视觉和交互细节问题。
- Definition of Done：
  - 主流程页面视觉一致、交互连贯。
  - 不引入额外功能范围。
- 验证方式：
  - 页面走查清单验证（首页、列表、详情、关于）。

### Milestone 6：后续增强方向（可选，不进入 V1）
- 目标：记录 V2 候选方向，不进入当前实现。
- 工作内容：
  - 记录评论、搜索、CMS、多语言、分析等可选项及触发条件。
  - 评估每个增强项的收益与复杂度。
- Definition of Done：
  - 形成明确的“后续决策清单”。
- 验证方式：
  - 可用于后续迭代评审，不影响 V1 交付。

## 5. 风险与边界（防跑偏策略）

### 风险 1：过早引入后端
- 问题：增加系统复杂度和运维成本，拖慢 MVP 交付。
- 避免方式：V1 严格使用本地内容文件，不引入服务端业务层。

### 风险 2：过度抽象
- 问题：代码结构复杂化，降低可读性和迭代速度。
- 避免方式：只做必要分层，不做插件系统和泛化架构。

### 风险 3：目录结构混乱
- 问题：职责不清，新增功能时容易交叉耦合。
- 避免方式：固定目录职责，新增文件必须符合分层规则。

### 风险 4：内容与页面耦合
- 问题：未来替换数据源时需要大规模改页面。
- 避免方式：页面禁止直接读取文件，统一走 `lib/posts`。

### 风险 5：为未来扩展提前实现复杂设计
- 问题：提前支付复杂度，当前收益小且风险高。
- 避免方式：只保留可扩展接口，不提前实现 CMS/API/DB。

## 6. 假设与执行边界

- 假设 1：当前仓库尚未初始化正式前端工程，现阶段以规划为先。
- 假设 2：V1 默认使用 MDX；若后续要求纯 Markdown，可在同一解析入口兼容。
- 假设 3：若出现“显著增加复杂度”的需求，将先评估并放入后续里程碑，不直接进入 V1。
- 执行边界：本文件只定义路径与标准，不执行完整功能开发。

---

## 建议起步顺序

建议从 **Milestone 1** 开始。  
原因：先建立清晰、稳定的项目骨架与分层边界，可以显著降低后续页面开发与数据链路实现时的返工概率，并确保后续每个里程碑都在同一套结构约束下推进。
